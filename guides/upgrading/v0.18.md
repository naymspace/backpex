# Upgrading to v0.18

## Bump Your Deps

Update Backpex to the latest version:

```elixir
defp deps do
  [
    {:backpex, "~> 0.18.0"}
  ]
end
```

## Filter System Refactoring (Breaking Change)

The filter system has been refactored to use Ecto changeset-based validation. This provides better type safety, validation, and a cleaner data flow. **This is a breaking change that affects custom filters.**

### What Changed

1. **New Filter Callbacks**: Filters now have `type/1` and `changeset/3` callbacks for validation
2. **Validated Query Values**: The `query/4` callback now receives validated, casted values instead of raw URL strings
3. **Removed `empty_filter_key`**: The internal `empty_filter_key` mechanism has been removed
4. **Changed `apply_filters` Signature**: In the Ecto adapter, `apply_filters/3` is now `apply_filters/4` with a new signature
5. **Removed `Backpex.LiveResource.filter_options/2`**: This function has been removed
6. **Changed Criteria Format**: The criteria keyword list now uses `filter_values` and `filter_configs` keys instead of `filters`

### Benefits

- **Type Safety**: Filter values are validated and cast before reaching your query functions
- **Better Error Handling**: Invalid filter values show inline validation errors in the UI
- **Cleaner Code**: No need to parse strings or handle invalid values in `query/4`
- **Consistent Behavior**: Invalid filters are silently ignored (show unfiltered results)

### Migrating Custom Filters

#### Before (v0.17)

```elixir
defmodule MyApp.Filters.CustomFilter do
  use BackpexWeb, :filter

  @impl Backpex.Filter
  def label, do: "My Filter"

  @impl Backpex.Filter
  def query(query, attribute, value, _assigns) do
    # value is a raw string from URL - you had to handle parsing
    case Integer.parse(value) do
      {int_value, ""} ->
        where(query, [x], field(x, ^attribute) > ^int_value)
      _ ->
        query  # Invalid value, return unchanged
    end
  end

  @impl Backpex.Filter
  def render(assigns), do: ~H"<%= @value %>"

  @impl Backpex.Filter
  def render_form(assigns) do
    ~H"""
    <input type="number" name={@form[@field].name} value={@value} />
    """
  end
end
```

#### After (v0.18)

```elixir
defmodule MyApp.Filters.CustomFilter do
  use BackpexWeb, :filter

  @impl Backpex.Filter
  def label, do: "My Filter"

  # NEW: Specify the Ecto type for your filter
  @impl Backpex.Filter
  def type(_assigns), do: :integer

  # OPTIONAL: Add custom validation
  @impl Backpex.Filter
  def changeset(changeset, field, _assigns) do
    Ecto.Changeset.validate_number(changeset, field, greater_than: 0)
  end

  @impl Backpex.Filter
  def query(query, attribute, value, _assigns) do
    # value is now already validated and cast to integer!
    # No need for parsing or error handling
    where(query, [x], field(x, ^attribute) > ^value)
  end

  @impl Backpex.Filter
  def render(assigns), do: ~H"<%= @value %>"

  @impl Backpex.Filter
  def render_form(assigns) do
    # @errors is now passed to render_form (defaults to [])
    ~H"""
    <input
      type="number"
      name={@form[@field].name}
      value={@value}
      class={["input input-sm mt-2", @errors != [] && "input-error bg-error/10"]}
    />
    <.error :for={msg <- @errors} class="mt-1">{msg}</.error>
    """
  end
end
```

### Migration Checklist

1. **Add `type/1` callback** to all custom filters
   - Return `:string` for single-value filters
   - Return `{:array, :string}` for multi-select filters
   - Return `:map` for range filters (with start/end values)
   - Return `:integer`, `:float`, etc. for numeric filters

2. **Remove value parsing** from `query/4`
   - Values are already cast to the correct type
   - Invalid values are filtered out before reaching `query/4`

3. **Optionally add `changeset/3`** for custom validation
   - Validate value ranges, formats, or allowed values
   - Use standard Ecto.Changeset validation functions

4. **Optionally display validation errors** in `render_form/1`
   - The `@errors` assign is now passed to `render_form/1` (defaults to `[]`)
   - Display errors below your filter inputs for better user feedback
   - Add error styling to inputs when errors are present

### Built-in Filter Updates

The built-in filters (`Backpex.Filters.Boolean`, `Backpex.Filters.Select`, `Backpex.Filters.MultiSelect`, `Backpex.Filters.Range`) have been updated automatically:

- **Boolean Filter**: Validates that selected keys exist in your `options/1`
- **Select Filter**: Validates that the selected value exists in your `options/1`
- **MultiSelect Filter**: Validates that all selected values exist in your `options/1`
- **Range Filter**: Validates date formats, number formats, and that start ≤ end

No changes required if you're only using built-in filters with `use Backpex.Filters.<Type>`.

### Type Reference

| Filter Type | `type/1` Return Value | Example Values |
|-------------|----------------------|----------------|
| Single value | `:string` | `"active"`, `"pending"` |
| Multiple values | `{:array, :string}` | `["a", "b", "c"]` |
| Range (dates/numbers) | `:map` | `%{"start" => "2024-01-01", "end" => "2024-12-31"}` |
| Numeric | `:integer` or `:float` | `42`, `3.14` |

### Validation Error Display

Invalid filter values now display inline validation errors in the filter dropdown. The filter is not applied to the query, showing unfiltered results instead.

The `@errors` assign is passed to your filter's `render_form/1` callback, containing a list of translated error messages for that filter field. Built-in filters automatically display these errors with appropriate styling. For custom filters, you can display errors using the `.error` component (available via `use BackpexWeb, :filter`):

```elixir
@impl Backpex.Filter
def render_form(assigns) do
  ~H"""
  <input
    type="text"
    name={@form[@field].name}
    value={@value}
    class={["input input-sm", @errors != [] && "input-error bg-error/10"]}
  />
  <.error :for={msg <- @errors} class="mt-1">{msg}</.error>
  """
end
```

For more details on the new filter validation system, see the [Filter Validation Guide](../filter/filter-validation.md).

### Adapter and Criteria Changes

If you're building a custom adapter or directly calling `Backpex.Adapters.Ecto.apply_filters/4`, note the following changes:

#### `apply_filters` Signature Change

**Before (v0.17):**

```elixir
# apply_filters/3 received a list of enriched filter maps
Backpex.Adapters.Ecto.apply_filters(query, filters, assigns)
# where filters = [%{field: :status, value: "active", filter_config: %{...}}, ...]
```

**After (v0.18):**

```elixir
# apply_filters/4 receives filter_values map and filter_configs keyword list separately
Backpex.Adapters.Ecto.apply_filters(query, filter_values, filter_configs, assigns)
# where filter_values = %{status: "active", count: 5}
# and filter_configs = [status: %{module: StatusFilter, ...}, count: %{module: CountFilter, ...}]
```

#### Criteria Format Change

The criteria keyword list format has changed:

**Before (v0.17):**

```elixir
criteria = [
  order: order,
  pagination: pagination,
  search: search_options,
  filters: LiveResource.filter_options(filter_values, filters)
]
```

**After (v0.18):**

```elixir
criteria = [
  order: order,
  pagination: pagination,
  search: search_options,
  filter_values: filter_values,
  filter_configs: filters
]
```

#### Removed Function

`Backpex.LiveResource.filter_options/2` has been removed. The filter config lookup now happens inside `apply_filters/4`.
